Протокол Агентского API Платформы BIL24 (Версия от 25.07.2025)

1.0 Введение и общие принципы

Агентский API платформы BIL24 представляет собой программный интерфейс, предназначенный для глубокой интеграции и взаимодействия с билетной системой. Его основная цель — предоставить разработчикам мощный инструмент для организации продажи билетов на культурные, спортивные и развлекательные мероприятия непосредственно из своих приложений, веб-сайтов или внутренних систем. Данный документ является исчерпывающим техническим руководством, описывающим все аспекты работы с API, от аутентификации до управления заказами.

Взаимодействие с API строится на следующих ключевых принципах:

* Формат запросов: Все запросы к API осуществляются методом HTTPS POST.
* Формат данных: Обмен данными происходит в формате JSON с использованием кодировки UTF-8.
* Сжатие данных: Для оптимизации трафика и ускорения получения больших объемов данных API поддерживает сжатие gzip. Для активации необходимо передать заголовок accept-encoding: gzip в вашем HTTPS запросе.

Сетевые адреса (Endpoints)

Для разработки и эксплуатации предусмотрены отдельные сетевые адреса (endpoints) для тестовой и реальной среды.

* BIL24:
  * Тестовая зона: https://api.bil24.pro:8443/json (резервный адрес на случай недоступности основного: https://api2.bil24.pro:8443/json)
  * Реальная зона: https://api.bil24.pro/json
* Kraftnum (рекомендовано для России):
  * Тестовая зона: https://api.kraftnum.ru:8443/json
  * Реальная зона: https://api.bil24.ru/json

В API используются следующие типы данных для полей в запросах и ответах:

Тип	Описание
int	Целое знаковое число в диапазоне от -2 147 483 648 до 2 147 483 647.
Uint	Целое беззнаковое число в диапазоне от 0 до 2 147 483 647.
long	Целое знаковое число в диапазоне от -9 223 372 036 854 775 808 до 9 223 372 036 854 775 807.
Ulong	Целое беззнаковое число в диапазоне от 0 до 9 223 372 036 854 775 807.

строка	Строковое значение в кодировке UTF-8 неограниченной длины.

2.0 Основы работы и Авторизация

Стратегически важно понимать сессионную модель, на которой построено взаимодействие с API. Корректная авторизация и управление сессиями являются ключом к доступу к защищенным методам, таким как резервирование мест и создание заказов. Без действительной сессии выполнение этих критически важных операций невозможно.

Сессионные и несессионные команды

Все команды протокола делятся на две категории: несессионные и сессионные. Несессионные команды предоставляют доступ к публичной информации (списки мероприятий, городов, мест проведения) и не требуют предварительной авторизации. Сессионные команды, напротив, оперируют данными конкретного пользователя (корзина, заказы, билеты) и требуют обязательной передачи идентификатора сессии sessionId.

Полный список несессионных команд:

1. GET_FILTER
2. GET_CITIES
3. GET_VENUE_TYPES
4. GET_VENUES
5. GET_ALL_ACTIONS
6. GET_ACTIONS_V2
7. GET_ACTION_EXT
8. GET_SEAT_LIST
9. GET_TICKETS_BY_DAY

Для вызова любой сессионной команды необходимо получить sessionId. Это достигается путем вызова команды AUTH (или CREATE_USER для интерфейсов типа "Билетная система"). Полученный sessionId является бессрочным. Рекомендуемый рабочий процесс: однократно вызвать AUTH (или CREATE_USER), получить userId и sessionId, безопасно сохранить эти данные на стороне клиента и использовать их для всех последующих сессионных запросов. Такой подход позволяет избежать ненужного создания пользователей и обеспечивает постоянную сессию.

Команды авторизации

Авторизация (command = AUTH)

Описание: Авторизация существующего пользователя или создание анонимного пользователя. При первом вызове (без userId и sessionId) система создает нового пользователя и возвращает его учетные данные. При последующих вызовах с этими данными происходит авторизация. Для интерфейсов "Билетный агент" и "Билетная система" следует использовать метод CREATE_USER.

Поля запроса

Поле	Тип	Обязательность	Описание
command	строка	Обязательное	AUTH
userId	Ulong	Обязательное, если есть sessionId	ID пользователя в системе.
sessionId	строка	Обязательное, если есть userId	ID сессии пользователя.

Состав полей ответа

Поле	Тип	Обязательность	Описание
userId	Ulong	Обязательное	ID пользователя в системе.
sessionId	строка	Обязательное	ID сессии пользователя.
frontendType	Uint	Обязательное	Тип интерфейса.

Пример запроса:

{
    "userId": 6104,
    "sessionId": "19c517c212bf173b44558308149db8ca",
    "command": "AUTH",
    "fid": 1185,
    "token": "2f1173e2b12fd5a111b6"
}


Пример ответа:

{
    "userId": 6104,
    "sessionId": "19c517c212bf173b44558308149db8ca",
    "command": "AUTH",
    "frontendType": 4,
    "resultCode": 0,
    "description": "OK"
}


После получения ответа, сохранить данные и использовать в запросах, где необходимы эти поля.

Создание пользователя (command = CREATE_USER)

Описание: Метод создает нового пользователя, который в дальнейшем может использоваться для резервирования мест, создания и оплаты заказа и получения билетов. Используется вместо метода AUTH для интерфейсов "Билетный агент" и "Билетная система".

Поля запроса

Поле	Тип	Обязательность	Описание
command	строка	Обязательное	CREATE_USER

Состав полей ответа

Поле	Тип	Обязательность	Описание
userId	Ulong	Обязательное	ID пользователя в системе.
sessionId	строка	Обязательное	ID сессии пользователя.

Пример запроса:

{
    "command": "CREATE_USER",
    "fid": 1184,
    "token": "24d3b97b64dfab187a5f"
}


Пример ответа:

{
    "userId": 18109,
    "sessionId": "d6035a755bd0862e713e03f337505bd0",
    "command": "CREATE_USER",
    "resultCode": 0,
    "description": "OK"
}


После получения ответа, сохранить данные и использовать в запросах, где необходимы эти поля.

Успешное получение sessionId открывает доступ ко всему функционалу API. Далее мы рассмотрим глобальные параметры, которые присутствуют в каждом запросе и ответе.

3.0 Глобальные параметры и структура ответов

Для обеспечения надежного и предсказуемого взаимодействия с API все запросы и ответы придерживаются стандартизированной структуры. Понимание этих общих элементов является обязательным для успешной интеграции.

Обязательные поля каждого запроса

Каждый POST запрос к API должен содержать в теле JSON следующие поля:

Поле	Тип	Обязательность	Описание
command	строка	Обязательное	Имя вызываемой команды (например, GET_CITIES).
fid	Ulong	Обязательное	Идентификатор вашего интерфейса (frontend ID), выдается при регистрации.
token	строка	Обязательное	Токен безопасности вашего интерфейса.
locale	строка	Необязательное	Предпочитаемый язык данных ответа в формате IETF BCP 47 (например, en-US). По умолчанию ru-RU.

Стандартная структура ответа

Каждый ответ от сервера имеет базовую JSON структуру, которая позволяет однозначно определить результат выполнения операции.

{
    "resultCode": 0,
    "description": "OK",
    "command": "COMMAND",
    ...
}


Общие поля, присутствующие в каждом ответе, описаны ниже:

Поле	Тип	Обязательность	Описание
resultCode	int	Обязательное	Код результата выполнения запроса. 0 означает успех.
description	строка	Обязательное	Текстовое описание результата. В случае ошибки содержит ее описание.
command	строка	Обязательное	Имя команды, которая была передана в запросе.
cause	строка	Необязательное	Дополнительная информация о причине ошибки (при resultCode не равном 0).

Важно: Если resultCode не равен 0, ответ будет содержать только указанные выше поля. Полезная нагрузка (данные команды) в этом случае отсутствует.

Значения поля resultCode

Значение	Описание
-1	Непредвиденная ошибка. Означает ошибку на стороне сервера (ошибка протокола, передачи данных и т.д.). Детали следует искать в поле description.
0	Успешное выполнение. Запрос обработан корректно, ответ содержит запрошенные данные.
1	Неверные идентификаторы сессии. Переданные userId или sessionId недействительны.
101	Стандартное сообщение для пользователя. Может использоваться для вывода информационных сообщений.
102	Почта не привязана. Возвращается в ответ на команду RESERVATION, если для операции требуется подтвержденный email.

Теперь, когда базовые принципы взаимодействия ясны, перейдем к детальному описанию всех доступных команд API.

Это предпочтительный метод для получения справочных данных, так как он гарантирует, что в пользовательском интерфейсе будут отображаться только релевантные опции для фильтрации.

4.4 Управление корзиной (Резервирование)

Резервирование мест является ключевым этапом перед оформлением заказа. Этот блок команд позволяет управлять жизненным циклом "корзины" пользователя: добавлять места в резерв, удалять их и просматривать текущее содержимое корзины. Резервирование является транзакционной операцией — в случае ошибки ни одно из запрошенных мест не будет зарезервировано.

Современный метод управления корзиной (RESERVATION)

Резервирование мест (type = RESERVE)

Устаревшие методы

Внимание! Описанные ниже методы являются устаревшими и не рекомендуются к использованию в новых интеграциях. Используйте современный метод RESERVATION с параметром type.

* RESERVE_BY_PLACE: Устаревший метод для резервирования мест с размещением по списку seatId.
* RESERVE (старая версия): Устаревший метод для резервирования мест без размещения через ассоциативный массив categoryQuantityMap.
* UN_RESERVE / UN_RESERVE_ALL (старая версия): Устаревший метод для снятия резерва, где тип операции передавался в поле type, а список мест для удаления — в поле seatList.

4.5 Управление заказами

После успешного резервирования мест в корзине, команды из этого блока позволяют перейти к следующему этапу — созданию заказа. Этот процесс включает в себя оформление, передачу данных для оплаты и, при необходимости, отмену заказа.

Статус заказа

Жизненный цикл заказа описывается набором статусов, которые возвращаются в полях statusExtStr (строковое представление) и statusExtInt (числовой код).

statusExtStr	statusExtInt	Описание
NEW	0	Новый заказ. Заказ создан и ожидает оплаты.
PROCESSING	1	Обработка. Оплата получена, идет процесс оформления билетов.
PAID	2	Завершен. Заказ успешно оплачен, билеты доступны.
CANCELLING	-1	Отмена. Идет процесс отмены неоплаченного заказа.
CANCELLED	-2	Отменен. Заказ отменен, зарезервированные места освобождены.

Статус заказа может меняться в следующих последовательностях:

* NEW → PROCESSING → PAID
* NEW → CANCELLING → CANCELLED

В исключительных случаях (например, из-за ошибок во внешних системах) возможен переход PROCESSING → CANCELLED. Статусы PAID и CANCELLED являются финальными. Получать билеты для пользователя следует только после того, как заказ перешел в статус PAID.
