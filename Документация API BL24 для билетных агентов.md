# **Техническая документация и руководство по интеграции с API платформы BIL24 для интерфейса "Билетный Агент"**

## **Введение**

Современная экосистема реализации билетов представляет собой сложный распределенный механизм, требующий высокой точности в управлении данными, транзакциями и сессиями пользователей. Платформа BIL24 предоставляет унифицированный программный интерфейс (API), который позволяет внешним системам взаимодействовать с единым пространством событий, залов и билетной массы. Данный отчет представляет собой исчерпывающее техническое руководство, разработанное специально для системных архитекторов и ведущих разработчиков, перед которыми стоит задача реализации клиентского приложения для роли "Билетный Агент" (Ticket Agent).

В отличие от поверхностных обзоров, данный документ погружается в архитектурные нюансы протокола BIL24, анализируя не только синтаксис команд, но и семантику взаимодействия компонентов. Мы рассмотрим жизненный цикл данных от момента получения списка событий до финализации финансовой транзакции, уделяя особое внимание вопросам отказоустойчивости, безопасности и консистентности данных. Основой для данного анализа послужила техническая документация платформы, а также результаты исследования поведения тестовых сред.1

Роль "Билетный Агент" в экосистеме BIL24 является ключевой: это независимый ритейлер, использующий собственные интерфейсы (веб-сайты, мобильные приложения, кассы) для продажи билетов из общего пула, предоставляемого Организаторами.2 Понимание этой роли критично для правильной реализации API: агент не управляет контентом (созданием событий), но несет полную ответственность за клиентский путь, резервирование мест и корректное проведение платежей.

## ---

**1\. Архитектурные принципы и транспортный уровень**

Взаимодействие с API BIL24 строится на принципах, отличающихся от классических RESTful сервисов. Платформа использует RPC-подобный (Remote Procedure Call) подход поверх протокола HTTP, где семантика действия определяется не HTTP-методом (GET/PUT), а полем command в теле запроса. Это требует от разработчика особого подхода к проектированию транспортного слоя приложения.

### **1.1. Протокол передачи данных**

Все взаимодействие с платформой осуществляется исключительно через протокол HTTPS с использованием метода POST. Это фундаментальное архитектурное решение, обеспечивающее инкапсуляцию всех параметров запроса, включая чувствительные данные сессии, внутри зашифрованного тела сообщения.1

* **Формат данных:** JSON (JavaScript Object Notation). Это стандарт де\-факто для современных веб\-сервисов, обеспечивающий легкость парсинга и читаемость.  
* **Кодировка:** UTF-8. Строгое соблюдение кодировки обязательно как для запросов, так и для ответов. Это критически важно для корректной передачи кириллических символов в названиях событий, городов и именах пользователей, что является стандартом для российского рынка, на котором оперирует платформа.1  
* **Сжатие данных:** Платформа поддерживает и рекомендует использование сжатия для оптимизации сетевого трафика. При выполнении "тяжелых" запросов, таких как получение списка всех мест в зале (GET\_SEAT\_LIST) или полного каталога событий, клиент должен передавать заголовок Accept-Encoding: gzip. Это позволяет получать ответы в сжатом виде, значительно сокращая время загрузки и нагрузку на канал связи.1

### **1.2. Сетевая топология и среды окружения**

Платформа BIL24 строго разделяет тестовые и продуктивные среды. Данные между этими зонами не синхронизируются, и попытка использовать идентификаторы (ID) из одной зоны в другой приведет к ошибкам. Для разработки приложения "Билетный Агент" необходимо настроить конфигурацию подключения, позволяющую переключаться между следующими точками входа (эндпоинтами).

#### **1.2.1. Тестовые зоны (Sandbox)**

Для этапа разработки и отладки платформа предоставляет несколько адресов. Использование порта 8443 является нестандартным для публичных веб\-API и может потребовать дополнительной настройки корпоративных фаерволов на стороне разработчика.1

| Зона | URL | Назначение |
| :---- | :---- | :---- |
| **BIL24 Test Primary** | https://api.bil24.pro:8443/json | Основной адрес для тестирования функционала. |
| **BIL24 Test Alt** | https://api2.bil24.pro:8443/json | Резервный адрес, используемый в случае блокировки основного или сетевых проблем. |
| **Kraftnum Test** | https://api.kraftnum.ru:8443/json | Тестовая зона платформы Kraftnum, полностью совместимой с протоколом BIL24, рекомендованная для использования в РФ. |

#### **1.2.2. Продуктивная зона (Production)**

После завершения тестирования приложение переключается на рабочий адрес. В этой среде происходят реальные финансовые транзакции.

* **URL:** https://api.bil24.pro/json.1

**Аналитический инсайт:** Наличие альтернативных URL и специфических портов указывает на высокую степень готовности платформы к работе в условиях нестабильной сетевой инфраструктуры. Разработчикам рекомендуется реализовать на уровне транспортного клиента механизм "failover" (автоматического переключения), который при недоступности основного хоста api.bil24.pro будет автоматически пробовать отправить запрос на api2.bil24.pro.

### **1.3. Структура конверта запроса**

Унификация — сильная сторона протокола BIL24. Каждый запрос, независимо от его функционального назначения (будь то поиск города или оплата заказа), должен содержать обязательный набор мета-данных. Это позволяет серверу однозначно идентифицировать инициатора запроса (интерфейс) и контекст безопасности.

Обязательные поля для любого запроса 1:

* **command** (String): Имя вызываемого метода API (например, GET\_VENUES). Регистрозависимое значение.  
* **fid** (Ulong): Идентификатор интерфейса (Frontend ID). Для тестового Билетного Агента используется значение 1271\.  
* **token** (String): Секретный ключ интерфейса, подтверждающий его подлинность.  
* **locale** (String, опционально): Предпочитаемый язык ответа в формате IETF BCP 47 (например, ru-RU). По умолчанию используется русский язык.

Пример структуры базового запроса:

JSON

{  
  "command": "GET\_VENUES",  
  "fid": 1271,  
  "token": "7c696b4af364928202dd",  
  "locale": "ru-RU"  
}

### **1.4. Структура конверта ответа**

Ответ сервера также стандартизирован. Клиентское приложение должно в первую очередь анализировать служебные поля конверта перед тем, как пытаться прочитать данные.

| Поле | Тип | Описание |
| :---- | :---- | :---- |
| **resultCode** | Int | Код результата операции. 0 означает успех. Любое другое значение сигнализирует об ошибке. |
| **description** | String | Текстовое описание результата (например, "OK" или текст ошибки). |
| **command** | String | Эхо-повтор имени команды из запроса. Полезно для логирования асинхронных ответов. |

Важное замечание по обработке ошибок:  
Если resultCode не равен 0, в ответе может присутствовать поле cause, содержащее технические детали ошибки. Приложение должно логировать содержимое cause для отладки, но пользователю следует показывать локализованное сообщение, основанное на коде ошибки или поле description.1

**2\. Идентификация и Управление Сессиями**

В отличие от публичных информационных запросов, операции, связанные с бронированием и покупкой билетов, требуют наличия пользовательского контекста. В терминологии BIL24 такие команды называются "сессионными" (sessional). Понимание механики сессий — критический этап интеграции, так как ошибки здесь приводят к невозможности совершить продажу.

### **2.1. Идентификация Интерфейса Агента**

Каждое приложение-агент идентифицируется парой fid и token. Эти данные являются статическими для конкретной инсталляции агента и выдаются администратором платформы (Сетевым Оператором).2

Для разработки и тестирования документация предоставляет следующие учетные данные для роли "Билетный Агент" 1:

* **FID:** 1271  
* **Token:** 7c696b4af364928202dd

**Анализ безопасности:** Токен интерфейса фактически является паролем приложения. Его нельзя передавать клиенту (в браузер или мобильное приложение) в открытом виде, если архитектура предполагает прямые вызовы API с клиента. Однако, учитывая структуру API, предполагается, что "Frontend" — это серверная часть агента, которая хранит токен в защищенном конфигурационном файле.

### **2.2. Пользовательские Сессии**

Сессия пользователя определяется парой userId (ID пользователя) и sessionId (ID сессии). Эти параметры обязательны для всех транзакционных команд (RESERVATION, CREATE\_ORDER, PAY\_ORDER).

Существует два механизма получения сессии, зависящих от типа интерфейса:

1. **AUTH (Авторизация):** Используется для интерфейсов типа "Браузер" (fid: 1185), где конечный пользователь вводит логин/пароль.  
2. **CREATE\_USER (Создание пользователя):** Используется для интерфейсов типа "Билетная система" (fid: 1184\) и, по логике работы, для "Билетных Агентов" (fid: 1271), работающих в режиме агрегатора.

#### **2.2.1. Специфика сессий Билетного Агента**

Для "Билетного Агента" часто используется модель, когда агент продает билеты анонимным покупателям или пользователям своей собственной системы регистрации. В этом случае Агент выступает как единый "супер-пользователь" или создает "теневых" пользователей в системе BIL24 для каждого своего клиента.

В предоставленных тестовых данных для fid: 1271 указаны статические идентификаторы пользователя:

* **UserId:** 33922  
* **SessionId:** 7c696b4af364928202dd

**Инсайт:** Обратите внимание, что для тестового агента sessionId совпадает с token. Это упрощение сделано только для тестовой среды. В реальной эксплуатации sessionId будет представлять собой уникальную строку, генерируемую сервером при создании пользователя или авторизации.

### **2.3. Жизненный цикл сессии**

Документация утверждает, что сессия пользователя действует "бессрочно" (indefinite).3 Это означает, что sessionId не истекает по тайм-ауту бездействия, как это бывает в банковских приложениях.

* **Стратегия хранения:** Приложение агента должно сохранить полученные userId и sessionId в надежном хранилище (база данных, зашифрованный кэш). При перезапуске сервера приложения необходимо переиспользовать эти данные, а не создавать нового пользователя.  
* **Обработка ошибки сессии:** Если сервер возвращает resultCode: 1 ("Session identifiers are wrong"), приложение должно считать текущую сессию недействительной. В этом случае необходимо запустить процедуру переавторизации (или создания нового пользователя) и повторить исходный запрос.3

**3\. Публичный API: Формирование Каталога Событий**

Первым шагом взаимодействия пользователя с системой агента является поиск и выбор события. Для этого используется группа "несессионных" команд, которые не требуют userId и sessionId. Эти команды возвращают публичные данные о географии, площадках и расписании.

### **3.1. Географическая навигация (GET\_CITIES, GET\_COUNTRIES)**

Для построения навигационного меню или фильтров используются справочники стран и городов.

* **GET\_COUNTRIES**: Возвращает список стран, где есть активные события.3  
* **GET\_CITIES**: Возвращает список городов.  
  * *Входные параметры:* Можно передать countryId для фильтрации городов конкретной страны.3  
  * *Ответ:* Содержит cityId, cityName и countryId.

**Рекомендация по кэшированию:** Эти данные являются условно-постоянными. Изменения в списке городов происходят крайне редко. Рекомендуется кэшировать ответ этих команд на стороне сервера Агента на срок от 24 часов до 1 недели, чтобы минимизировать нагрузку на API и ускорить отклик интерфейса.

### **3.2. Площадки и Залы (GET\_VENUES)**

События привязаны к физическим местам проведения (Venues).

* **Команда:** GET\_VENUES  
* **Параметры запроса** 3:  
  * cityId (Ulong, необязательно): Фильтр по городу.  
  * venueTypeId (Uint, необязательно): Фильтр по типу площадки (Театр, Музей, Стадион и т.д.). Типы площадок можно получить командой GET\_VENUE\_TYPES.3  
* **Структура ответа** 3:  
  * venueId (Ulong): Уникальный ID площадки.  
  * venueName (String): Название (например, "Большой Театр").  
  * address (String): Физический адрес.  
  * latitude, longitude: Географические координаты для отображения на карте.

### **3.3. Получение списка событий (GET\_ACTIONS\_V2 и GET\_ALL\_ACTIONS)**

Основной единицей контента является "Action" (Событие/Спектакль). Важно различать само событие (название, описание) и его сеансы (конкретные даты и время).

#### **3.3.1. GET\_ACTIONS\_V2**

Стандартный метод получения списка событий.

* **Особенности:** Возвращает список актуальных событий.  
* **Поля ответа** 3:  
  * actionId: ID события (группирует несколько сеансов).  
  * name: Название.  
  * genreList: Список жанров, к которым относится событие.3  
  * minPrice / maxPrice (или maxSum): Диапазон цен для отображения "от... до...".

#### **3.3.2. GET\_ALL\_ACTIONS**

Специализированный метод, изначально предназначенный для Билетных Систем, но крайне полезный для Агентов.

* **Преимущество:** Он возвращает **полный дамп** данных, включая информацию, обычно получаемую через GET\_FILTER. Это позволяет за один запрос получить всю структуру событий, сеансов и площадок.3  
* **Стратегия синхронизации:** Для высоконагруженных Агентов рекомендуется использовать GET\_ALL\_ACTIONS в фоновом режиме (например, раз в 15-60 минут) для синхронизации локальной базы данных с платформой BIL24. Это позволит реализовать мгновенный поиск и фильтрацию событий на стороне Агента без задержек, связанных с сетевыми запросами к API при каждом действии пользователя.

**Важное уточнение:** Для бронирования билета ключевым идентификатором является не actionId, а actionEventId (ID конкретного сеанса события), который содержится внутри структуры данных события.

## ---

**4\. Работа с Инвентарем: Схемы Залов и Места**

После выбора события пользователь переходит к выбору мест. Это самый ресурсоемкий этап взаимодействия, требующий работы с большими массивами данных.

### **4.1. Доступность Мест (GET\_SEAT\_LIST)**

Эта команда возвращает статус каждого места в зале для выбранного сеанса.

* **Тип:** Несессионная команда (можно вызывать без авторизации пользователя).3  
* **Запрос:** Должен содержать actionEventId (ID сеанса) и fid/token.  
* **Ответ:** Массив объектов, описывающих места. Для стадионов размер массива может достигать десятков тысяч элементов.

Типичная структура объекта места (реконструирована на основе данных о бронировании 3):

* seatId (Ulong): Уникальный идентификатор места. Используется для бронирования.  
* price (Cur): Стоимость.  
* sector (String): Название сектора (например, "Партер").  
* row (String): Ряд.  
* number (String): Номер места.  
* status: Состояние места (свободно, занято, забронировано). *Примечание: точное имя поля статуса в документации не приведено, но оно логически необходимо.*  
* categoryPriceId: ID ценовой категории (важно для входных билетов без мест).

**Оптимизация:** Обязательно используйте сжатие gzip. На мобильных устройствах парсинг такого JSON может занимать время, поэтому рекомендуется выполнять его в фоновом потоке UI.

### **4.2. Визуализация Зала (GET\_SCHEMA)**

Для отображения интерактивной схемы зала используется команда GET\_SCHEMA (или GET\_SEAT\_COORDINATES).

* **Тип:** Сессионная команда (требует sessionId).1  
* **Функционал:** Возвращает координаты мест для отрисовки на графической подложке (схеме зала).  
* **Связывание данных:** Приложение Агента должно сопоставить данные из GET\_SEAT\_LIST (цены и статусы) с данными из GET\_SCHEMA (координаты X/Y) по ключу seatId. Это позволяет отрисовать карту, где место отображается цветом, соответствующим его цене или статусу.3

## ---

**5\. Транзакционный Цикл: Бронирование и Продажа**

Ядром функционала Билетного Агента является проведение сделки. В архитектуре BIL24 этот процесс разбит на атомарные шаги, управляемые конечным автоматом состояний заказа.

### **5.1. Шаг 1: Резервирование (RESERVATION)**

Перед созданием заказа места должны быть временно заблокированы за пользователем. Это предотвращает "double booking" (двойную продажу) одних и тех же мест разным пользователям.

* **Команда:** RESERVATION  
* **Тип:** Сессионная.  
* **Параметры запроса** (для блокировки) 1:  
  * actionEventId: ID сеанса.  
  * seatList: Массив ID мест (seatId) для мест с рассадкой.  
  * categoryList: Массив категорий (categoryPriceId) и количества (quantity) для входных билетов без мест.  
* **Ответ:**  
  * seatList: Список успешно зарезервированных мест.  
  * cartTimeout (Uint): **Критически важный параметр.** Время жизни брони в секундах (например, 600 секунд).  
  * totalSum: Общая сумма корзины.

**Механика Тайм-аута:** Агент обязан отобразить пользователю таймер обратного отсчета. Если пользователь не успевает оформить заказ за время cartTimeout, сервер автоматически освобождает места. Агент должен корректно обработать это событие, очистив корзину в интерфейсе.

Для отмены резерва (например, если пользователь убрал билет из корзины) используется та же команда RESERVATION, но со специальным типом действия или передачей списка мест для разблокировки (в документации упоминается UN\_RESERVE как тип ответа при отмене 3).

### **5.2. Шаг 2: Создание Заказа (CREATE\_ORDER)**

Когда корзина сформирована, пользователь переходит к оформлению. На этом этапе резерв превращается в "Заказ" (Order).

* **Команда:** CREATE\_ORDER (или CREATE\_ORDER\_EXT для билетных систем).3  
* **Обязательные данные:** userId, sessionId.  
* **Дополнительные данные** (изменения протокола 3):  
  * email (String): Email пользователя. Обязателен для отправки электронных билетов.  
  * phone (String): Телефон.  
  * paymentType: Тип оплаты.  
  * deliveryType: Тип доставки (обычно "Электронный билет").  
* **Ответ:**  
  * orderId (Ulong): Уникальный номер заказа.  
  * status: Начальный статус заказа (обычно NEW или код 0/1).1  
  * paymentLink: Ссылка на оплату (если используется платежный шлюз платформы).

**Инсайт:** Метод CREATE\_ORDER\_EXT рекомендуется для "Билетных систем" 3, что подразумевает расширенные возможности передачи метаданных. Для Агента (fid: 1271\) стандартным, вероятно, является CREATE\_ORDER, но стоит проверить доступность EXT версии, если требуются сложные параметры.

### **5.3. Шаг 3: Оплата Заказа (PAY\_ORDER)**

Финальный шаг, фиксирующий продажу. Для Билетного Агента это действие имеет финансовые последствия: баланс агента уменьшается на сумму заказа (за вычетом комиссии), либо формируется задолженность перед Оператором.

* **Команда:** PAY\_ORDER.3  
* **Условие:** Применима только к заказам в статусе NEW.  
* **Параметры:**  
  * orderId: ID заказа.  
  * userId, sessionId.  
* **Ответ:**  
  * statusExtStr: PAID.  
  * statusExtInt: 2 (Код статуса "Оплачен").1

**Варианты реализации оплаты:**

1. **Прямая интеграция:** Агент принимает деньги от клиента через свой эквайринг, затем вызывает PAY\_ORDER, оплачивая билет со своего счета в системе BIL24.  
2. **Редирект:** Агент перенаправляет пользователя на шлюз BIL24 (используя ссылку из ответа CREATE\_ORDER), и платформа сама вызывает смену статуса после успешной оплаты. В этом случае Агент должен опрашивать статус заказа (GET\_ORDER\_INFO) для подтверждения успеха.

### **5.4. Отмена Заказа (CANCEL\_ORDER)**

Если пользователь отказался от оплаты или транзакция не прошла, Агент **обязан** вызвать CANCEL\_ORDER. Это правило "хорошего тона" в распределенной системе: оно немедленно возвращает билеты в продажу, не дожидаясь истечения тайм-аута корзины.1

**6\. Спецификация Типов Данных и Особенности Реализации**

Разработка надежного клиента требует строгого соблюдения типов данных, определенных в протоколе.

### **6.1. Проблема Ulong и JavaScript**

Многие идентификаторы в системе (fid, userId, seatId, orderId) имеют тип Ulong (Unsigned 64-bit Integer).

* **Диапазон:** от 0 до 18,446,744,073,709,551,615.  
* **Риск:** Стандартный тип Number в JavaScript (используемый в Node.js и браузерах) является 64-битным числом с плавающей точкой (IEEE 754). Он может точно представлять целые числа только до 2^53 \- 1 (9,007,199,254,740,991).  
* **Проблема:** ID, превышающие этот лимит (что вполне реально для глобальных ID билетов), будут округлены и искажены при парсинге JSON. Например, ID 9223372036854775807 может превратиться в 9223372036854776000\.  
* **Решение:** При работе с JSON в среде JavaScript необходимо использовать специальные библиотеки (например, json-bigint) или обрабатывать все поля \*Id как **Строки (String)**, а не числа. В строго типизированных языках (Java, C\#, Go) следует использовать типы uint64 или BigInteger.

### **6.2. Денежный формат (Cur)**

Тип Cur представляет собой число с фиксированной точкой (до 4 знаков после запятой, хотя обычно используются 2).

* **Рекомендация:** При выполнении арифметических операций (сложение стоимости билетов в корзине) избегайте накопления ошибок округления, свойственных числам с плавающей точкой. Рекомендуется оперировать "копейками" (целыми числами) или использовать тип Decimal.1

### **6.3. Обработка Ошибок (resultCode)**

Приложение Агента должно реализовывать централизованный обработчик кодов ответов.

| Код | Значение | Действие Агента |
| :---- | :---- | :---- |
| 0 | Успех | Обработать payload ответа. |
| \-1 | Системная ошибка | Ошибка протокола, БД или сети. Реализовать повтор запроса (retry) с экспоненциальной задержкой. |
| 1 | Ошибка сессии | Невалидный userId/sessionId. Не показывать ошибку пользователю. Фоном выполнить переавторизацию и повторить запрос. |
| 101 | Сообщение пользователю | Показать текст из поля description пользователю (например, "Место уже занято"). |
| 102 | Email не привязан | Запросить ввод Email (актуально при бронировании).3 |

**7\. Стратегия Разработки Приложения Билетного Агента**

Для успешной интеграции рекомендуется придерживаться следующей архитектурной стратегии:

### **7.1. Слой Абстракции API (Connector Layer)**

Не вызывайте методы API напрямую из кода контроллеров UI. Создайте изолированный класс/модуль Bil24Connector.

* Он должен инкапсулировать fid и token.  
* Он должен автоматически добавлять их в каждый запрос.  
* Он должен прозрачно обрабатывать gzip.  
* Он должен содержать логику "Circuit Breaker" (прерывателя цепи) при недоступности API.

### **7.2. Локальное Зеркалирование (Local Mirroring)**

Не полагайтесь на GET\_ACTIONS\_V2 для каждого просмотра страницы пользователем.

1. Создайте фоновый процесс (Worker), который раз в час (или чаще) вызывает GET\_ALL\_ACTIONS.  
2. Сохраняйте полученные события и площадки в собственную базу данных (SQL/NoSQL).  
3. Поиск, фильтрацию и отображение афиши стройте на базе **локальных данных**.  
4. К API обращайтесь только за динамическими данными: статусом мест (GET\_SEAT\_LIST) и бронированием.

### **7.3. Идемпотентность и Очереди**

При реализации оплаты (PAY\_ORDER) учитывайте риск сетевого сбоя. Если вы списали деньги с карты клиента, но не получили ответ от PAY\_ORDER:

* Не считайте транзакцию проваленной сразу.  
* Поставьте задачу в очередь на повторную проверку статуса заказа (GET\_ORDER\_INFO).  
* Если заказ остался NEW, попробуйте вызвать PAY\_ORDER повторно.

Ключ к успеху лежит в строгом соблюдении протокола управления сессиями, корректной обработке 64-битных идентификаторов и реализации надежных механизмов синхронизации данных.
